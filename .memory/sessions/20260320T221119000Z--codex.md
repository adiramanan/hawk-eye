## Context

- Reviewed the repo memory files and traced the current authorization failure to client-bound save capabilities surviving source writes poorly across HMR reconnects.
- The likely failure mode is: a capability is issued to one websocket client, a source write triggers HMR/reconnect, and the next save arrives from a different client object with the same capability string.

## Actions

- Updated the plugin server state to track issued save capabilities in a process-local set in addition to the per-client weak map.
- Changed save-capability validation so a previously issued capability remains valid even if the websocket client identity changes after HMR/source refresh.
- Added a save-handler regression test covering the reconnect case where one client receives the capability and a different client sends the save request with the same token.
- Verified the fix with `pnpm test -- tests/save-handler.test.ts` and `pnpm type-check`.

## Decisions

- Save capabilities should survive websocket client identity changes caused by HMR reconnects, while still requiring an issued random token and the existing `enableSave` gate.

## Open Threads

- Server restarts still invalidate all prior capabilities because the in-memory issued-capability set is rebuilt on startup; if that becomes user-visible, add a client-side automatic re-auth retry path on `invalid-capability`.

## Summary

Fixed save-capability authorization failures across HMR reconnects by validating issued capability tokens independently of the transient websocket client object.
