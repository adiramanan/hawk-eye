## Context

- Reviewed `.memory/CURRENT_CONTEXT.md`, `.memory/MEMORY.md`, `.memory/BLOCKERS.md`, and `.memory/PHASE_STATUS.md` on branch `dev`.
- Continuing from the dynamic `className` alpha fallback work to support the remaining blocked `style={expr}` persistence case.

## Actions

- Added a rewritable `expression` style-attribute state in the source writer and changed non-detached saves to wrap `style={expr}` as `style={{ ...expr, persistedProp: value }}` before applying inline mutations.
- Kept detached mode on the previous behavior by disallowing style-expression wrapping there.
- Switched the client from a blocked dynamic-class + dynamic-style state to a non-blocking `style-wrap-fallback` hint so `Update Design` stays enabled.
- Updated writer, save-handler, and DesignTool tests to cover identifier-expression wrapping, call-expression wrapping, repeated saves without re-wrapping, and unsupported non-JSX style attributes.
- Verified the change set with `pnpm test -- tests/source-writer.test.ts tests/save-handler.test.ts tests/design-tool.test.ts` and `pnpm type-check`.

## Decisions

- Dynamic `style={expr}` persistence is now an alpha-supported save path for non-detached edits; Hawk-Eye preserves the original expression and appends persisted declarations after a spread so saved edits win.
- The public style-analysis payload remains unchanged; the client derives the new wrap hint from the existing `classAttributeState` and `styleAttributeState` fields.
- Truly unsupported style attributes still fail through warning-derived save errors rather than the generic no-op writer message.

## Open Threads

- Dynamic style attributes that are not JSX expressions, such as string-literal `style="..."`, still cannot be rewritten and remain save errors.

Implemented dynamic `style={expr}` wrapping for editable dynamic `className` saves, removed the client-side block for that case, and covered the new writer contract with focused tests plus a clean workspace type-check.
