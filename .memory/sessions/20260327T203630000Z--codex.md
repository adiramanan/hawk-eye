## Context

- Investigated a border-specific inconsistency where grouping stroke widths from a bottom-only border could leave the bottom stroke on a different color source until the user manually changed the border color field.

## Actions

- Added border-color normalization during border-width draft updates in `packages/client/src/DesignTool.tsx`.
- When grouped stroke-width edits expose new sides, the draft now seeds `borderColor` from the currently visible border side if the generic border color disagrees.
- Added a regression test covering the bottom-only colored border case and reran the full focused design-tool suite.

## Decisions

- Width edits that turn a one-sided border into a multi-sided border should preserve the currently visible stroke color as the single source of truth.
- This normalization belongs in the draft-update layer, because the issue is caused by the interaction between grouped width edits and mixed border-color state.

## Open Threads

- The focused design-tool coverage now includes this grouped stroke-color normalization path, but there is still no separate low-level control test for mixed border-color state outside the full inspector flow.

## Summary

Grouped stroke-width edits now preserve the visible border color when expanding a one-sided stroke to all sides, so the user no longer has to touch the color field just to make the fourth side match.
