## Context

- The class-target dropdown previously changed only `activeClassTargetId`, leaving the existing dirty property snapshots untouched.
- With that model, editing one authored class target and then switching the dropdown caused the same pending changes to be previewed and saved against the newly selected selector, which also updated peer elements that shared that class.

## Actions

- Updated `changeClassTarget()` to clear the current preview overrides and rebuild a fresh selection draft for the newly selected class target instead of carrying over the prior dirty property values.
- Kept a fallback path that only swaps `activeClassTargetId` if the inspectable element cannot be resolved.
- Added a regression test that edits one class target, switches to a second target on the same element, and verifies the old preview rule does not jump to the second selector or its peers.
- Verified the fix with `pnpm vitest --run tests/design-tool.test.ts tests/smoke.test.ts tests/ws-server.test.ts`.

## Decisions

- Treat class-target switching as a rebase action: unsaved edits belong to the currently edited class target and should not migrate when the user switches the dropdown to inspect or edit a different class.

## Open Threads

- The current behavior resets pending class-target edits on target switch rather than preserving multiple unsaved target-specific edit sets at once.

## Summary

Switching the authored class target dropdown no longer rebinds pending edits to the new selector, so related elements stop inheriting the previous class's unsaved preview changes.
