## Context

- The demo chip/status-chip CSS was already decoupled, but Hawk-Eye could still preview or save against the old combined selector if the dev server had cached authored class targets before the stylesheet edit.
- The existing cache invalidation only ran after Hawk-Eye's own save path modified files, so manual CSS edits or external formatter changes could leave stale class-target selectors active until a server restart.

## Actions

- Added `registerStyleFileInvalidation()` and `invalidateStyleCachesForFile()` so watched `.css`, `.scss`, `.sass`, and `.less` changes inside the project root clear both the authored class-target index and the style-analysis cache.
- Wired that watcher registration into the Vite plugin `configureServer()` path before the inspect/save handlers are registered.
- Updated the smoke tests to stub the Vite watcher alongside the websocket bridge.
- Added a regression test proving that after an external stylesheet edit, the same selected element continues to report the stale combined selector until invalidation runs, then refreshes to the new dedicated `.he-chip` selector.
- Verified the focused suite with `pnpm vitest --run tests/smoke.test.ts tests/ws-server.test.ts`.

## Decisions

- Treat external stylesheet edits the same as Hawk-Eye-authored stylesheet saves for cache freshness: both should invalidate the authored class-target index and the cached style-analysis payloads.
- Limit watcher-driven invalidation to stylesheet file extensions inside the resolved workspace root so unrelated file changes do not churn the class-target cache unnecessarily.

## Open Threads

- The running dev server still needs to receive this updated plugin code; if it was started before the patch, restart it once so the new watcher hooks take effect.

## Summary

Hawk-Eye now refreshes cached class-target selectors when watched stylesheet files change, so external CSS edits no longer require a dev-server restart to stop stale selectors from affecting preview/save behavior.
