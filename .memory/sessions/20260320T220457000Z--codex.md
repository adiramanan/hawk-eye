## Context

- Reviewed the repo memory files and traced the current save failure to stale `data-hawk-eye-source` handling in the client draft lifecycle.
- The likely failure mode is: save succeeds, source/HMR updates the DOM token, but the cached instance key and draft `file:line:column` remain stale for the next save.

## Actions

- Updated `createInspectableElementKey` to invalidate cached keys when an element's `data-hawk-eye-source` token changes, instead of reusing a permanently stale key.
- Fixed `mergeSelectionDraft` so refreshed selection details carry the latest `source` string forward with the updated `file:line:column`.
- Added a selected-element source rebinding path in `DesignTool.tsx` so the locked draft rekeys itself and requests fresh selection metadata when the DOM token changes after save/HMR refresh.
- Added a mutation observer for the locked selection to watch source-attribute and subtree changes even when there are no dirty drafts pending.
- Added a regression test covering save -> source-token change -> second save with refreshed coordinates, and verified with `pnpm test -- tests/design-tool.test.ts` plus `pnpm type-check`.

## Decisions

- Treat post-save source-token changes as a client-state rebind problem, not a writer-path problem; the selected draft must follow the DOM token after HMR/source refresh.
- Keep the fix local to the client selection lifecycle rather than changing the save payload contract.
