## Context

- Investigated a mismatch where the initialized stroke color field showed raw `oklch(...)` text but picker interactions rewrote the value as hex.
- The color input already normalizes parsed colors to hex, so the issue reduced to missing OKLCH parsing support in the shared color utility.

## Actions

- Added direct `oklch(...)` parsing and conversion to RGBA in `packages/client/src/utils/color.ts`.
- Added a regression test that seeds a border color snapshot with an OKLCH string and verifies the visible input normalizes to hex.
- Re-ran the focused design-tool suite and the targeted new regression.

## Decisions

- The UI should normalize supported CSS color formats, including OKLCH, into the same hex display used elsewhere in the property controls.
- The fix belongs in the shared parser rather than special-casing the stroke control.

## Open Threads

## Summary

Stroke and other color inputs can now parse initialized OKLCH values and render them in hex immediately, so the field no longer flips formats only after the picker is used.
